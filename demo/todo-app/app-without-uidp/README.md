# UIDPなし実装: シンプルToDo

`comparison.md` の図（画面遷移図、ワイヤーフレーム、機能要件）のみを参照して実装したReactアプリケーション。

## 起動方法

```bash
npm install
npm run dev
```

## 参照した情報

1. 画面遷移図（TodoList ↔ TodoDetail）
2. ワイヤーフレーム（一覧画面、詳細画面、編集モード）
3. 機能要件（箇条書き）

## UIDP版との違い

### 1. URL設計

| 項目 | UIDPなし | UIDP版 |
|------|---------|--------|
| 一覧画面 | `/` | `/todos` |
| 詳細画面 | `/?id=xxx` | `/todos/:id` |

**理由**: 図にはURL設計が明記されていなかったため、クエリパラメータ方式を選択。

### 2. タスク追加位置

| 項目 | UIDPなし | UIDP版 |
|------|---------|--------|
| 追加位置 | 末尾 | 先頭（`position: beginning`） |

**理由**: 図には追加位置の指定がなかったため、デフォルトの末尾追加。

### 3. ID生成

| 項目 | UIDPなし | UIDP版 |
|------|---------|--------|
| ID生成 | `Date.now().toString()` | `crypto.randomUUID()` |

**理由**: 図にはID生成方法の指定がなかったため、簡易的なtimestamp方式。

### 4. 削除確認ダイアログ

| 項目 | UIDPなし | UIDP版 |
|------|---------|--------|
| ダイアログ | `window.confirm()` | MUI Dialog |
| タイトル | なし | 「タスクの削除」 |
| 確認ボタン | 「OK」 | 「削除」 |

**理由**: 要件に「確認ダイアログ付き」とあったが、デザインの指定がなかったため`window.confirm`を使用。

### 5. 日付型

| 項目 | UIDPなし | UIDP版 |
|------|---------|--------|
| 型 | `Date` | `string` (ISO8601) |

**理由**: 図には型の指定がなく、localStorageでの復元時に変換処理が必要になった。

### 6. 空の場合の表示

| 項目 | UIDPなし | UIDP版 |
|------|---------|--------|
| 空表示 | なし | 「タスクがありません」 |

**理由**: 図には空の状態のワイヤーフレームがなかったため未実装。

### 7. トースト通知

| 項目 | UIDPなし | UIDP版 |
|------|---------|--------|
| 保存時 | なし | 「保存しました」 |
| 削除時 | なし | 「削除しました」 |

**理由**: 図にはトースト通知の記載がなかったため未実装。

### 8. イベント伝播制御

| 項目 | UIDPなし | UIDP版 |
|------|---------|--------|
| チェックボックス | 伝播あり | `bubble: false`で停止 |
| 削除ボタン | 伝播あり | `bubble: false`で停止 |

**理由**: 図にはイベント伝播の指定がなく、実装時に問題が発覚するパターン。

## 結論

ワイヤーフレームと要件だけでは、以下の情報が不足する：

- **URL設計**: SEO、ブックマーク、履歴管理に影響
- **データ構造**: 型定義、初期値、永続化方法
- **UI詳細**: 空状態、エラー状態、ローディング状態
- **UX詳細**: 追加位置、通知、確認ダイアログのデザイン
- **イベント制御**: 伝播、バブリング

これらは実装者の「解釈」に委ねられ、意図と異なる実装になるリスクがある。
